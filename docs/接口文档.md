# 1. 文档目标

本文根据 `原型图.pdf` 的 UI 模块与 `设计文档.md` 的产品/架构内容，定义移动端与后端的接口契约，覆盖：

```text
场景选择页
语音练习房间
实时字幕与纠错浮层
对话报告页
历史记录页
用户画像页
设置页
流量控制面板
实时音频 WebSocket 协议
后端到豆包/腾讯智聆/TTS/ASR 的 Adapter 边界
```

核心设计原则：

| 原则 | 说明 |
|---|---|
| App 不直连供应商 | 腾讯智聆 SecretId/SecretKey、豆包 API Key、TTS 密钥全部只在后端保存 |
| 快慢链路拆分 | AI 首声走快链路；发音测评和详细纠错走旁路慢链路 |
| 事件驱动 | 语音房间用 WebSocket 传输音频、字幕、AI 回复、纠错、评分、状态 |
| 可观测 | 所有模型/语音调用必须记录 latency、cost、provider、requestId |
| 可降级 | 豆包、TTS、智聆异常时返回可恢复状态，不直接中断整场练习 |

## 2.2 WebSocket API

| 通道 | 路径 | 用途 | 前端显示策略 |
|---|---|---|---|
| Realtime Dialogue | `/ws/v1/practice-sessions/{sessionId}/dialogue` | 主对话通道：音频分片、实时字幕、AI 回复、TTS 音频、状态同步 | 必须显示 |
| Pronunciation Assessment | `/ws/v1/practice-sessions/{sessionId}/pronunciation` | 发音测评旁路：词级问题、音标、发音/流利度/完整度分数、发音提示 | 可关闭显示，后台仍处理 |
| Grammar Correction | `/ws/v1/practice-sessions/{sessionId}/correction` | 语法/表达纠错旁路：quickFix、解释、严重程度、打断建议 | 可关闭显示，后台仍处理 |

说明：前端可以在练习房间内关闭发音提示、语法提示或两者都关闭。关闭只影响 UI 浮层和房间内提示展示；后端仍会执行发音测评和语法/表达纠错，并将结果写入报告、画像和后续推荐的数据源。

## 2.3 后端 Adapter 接口

| Adapter | 方向 | 用途 |
|---|---|---|
| `DialogueModelAdapter` | Backend -> 豆包 | 实时对话、语法纠错、报告、画像、推荐 |
| `PronunciationAssessmentAdapter` | Backend -> 腾讯智聆 | 发音测评、流利度、完整度、词级问题 |
| `TtsAdapter` | Backend -> TTS | AI 文本转语音 |
| `AsrAdapter` | Backend -> ASR/豆包多模态 | 用户语音转写 |
| `MemoryAdapter` | Backend -> DB/Markdown/Vector DB | 热记忆、冷记忆、用户档案 |
| `RateLimitAdapter` | Backend -> Redis/DB | 配额、限流、降级 |


## 3.2 鉴权

移动端 REST 请求：

```http
Authorization: Bearer <accessToken>
X-Client-Platform: ios | android
X-App-Version: 1.0.0
X-Request-Id: uuid
```

WebSocket 连接：

```text
wss://api.example.com/ws/v1/practice-sessions/{sessionId}?token=<accessToken>&clientSeq=<number>
```

约束：

```text
accessToken 过期时，REST 返回 401。
WebSocket token 过期时，服务端发送 auth_expired 事件后关闭连接。
供应商密钥只允许后端读取，禁止下发 App。
```

# 8. 实时 WebSocket 协议

实时房间使用 3 个 WebSocket 通道。主对话通道保证 coach 对话体验，发音测评和语法纠错走旁路通道异步返回。

```text
/dialogue       用户音频、字幕、coach 回复、TTS 音频、session_state、error
/pronunciation 用户 turn 音频测评结果、词级问题、音标、score_snapshot
/correction    用户最终转写文本的语法/表达纠错、quickFix、explanation、severity
```

前端显示开关：

```text
pronunciationVisible=false 时，不展示 pronunciation_hint 浮层，但后端仍处理并落库。
grammarVisible=false 时，不展示 grammar_hint 浮层，但后端仍处理并落库。
两个开关都关闭时，主对话仍正常进行，报告仍包含发音和语法数据。
```

```text
wss://api.example.com/ws/v1/practice-sessions/{sessionId}/dialogue?token=<accessToken>&clientSeq=1
wss://api.example.com/ws/v1/practice-sessions/{sessionId}/pronunciation?token=<accessToken>&clientSeq=1
wss://api.example.com/ws/v1/practice-sessions/{sessionId}/correction?token=<accessToken>&clientSeq=1
```

客户端 -> 服务端：

```json
{
  "type": "audio_chunk",
  "clientSeq": 12,
  "sessionId": "ps_001",
  "sentAt": "2026-06-05T09:42:03.120Z",

}
```

服务端 -> 客户端：

```json
{
  "type": "transcript_delta",
  "serverSeq": 18,
  "sessionId": "ps_001",
  "createdAt": "2026-06-05T09:42:03.260Z",

```

要求：

```text
clientSeq 单调递增，用于乱序诊断。
serverSeq 单调递增，用于 App 侧重放保护。
payload 按事件类型定义。
二进制音频优先使用 binary frame；若框架限制，可使用 base64。
```

### 8.3.2 音频分片

JSON base64 模式：

```json
{
  "type": "audio_chunk",
  "clientSeq": 12,
  "sessionId": "ps_001",
  "payload": {
    "turnClientId": "cturn_001",
    "seq": 12,
    "format": "pcm_s16le",
    "sampleRate": 16000,
    "channels": 1,
    "durationMs": 40,
    "data": "base64_pcm_1280_bytes"
  }
}
```

Binary 模式：

```text
Binary frame payload = 1280 bytes PCM
Metadata 每 1 秒或每个 turn_start 发送一次 audio_meta 事件。
```

智聆对齐要求：

```text
采样率：16000Hz
采样精度：16bit
声道：mono
格式：pcm 优先
发送节奏：每 40ms 发送 40ms 音频
16k PCM 40ms 分片大小：1280 bytes
音频发送速率不得超过 1:1 实时率
音频包间隔超过 6 秒可能导致腾讯智聆出错
```

# 9. 实时链路编排

## 9.1 快链路：首声优先

```text
App audio_chunk
  -> Backend Realtime Gateway
  -> ASR / 豆包多模态输入
  -> 豆包生成 first speakable chunk
  -> TTS 首包
  -> ai_audio_chunk 返回 App 播放
```

SLA：

| 指标 | 目标 |
|---|---:|
| AI 首次发声 P50 | < 1.5s |
| AI 首次发声 P90 | < 2s |
| LLM first speakable chunk P90 | < 1.2s |
| TTS 首包 P90 | < 800ms |

## 9.2 慢链路：发音与纠错

```text
用户音频片段
  -> 后端缓冲/切 turn
  -> 腾讯智聆 WSS
  -> 解析 PronAccuracy / PronFluency / PronCompletion / Words
  -> 豆包生成教学化解释
  -> pronunciation_hint 返回 App
```

SLA：

| 指标 | 目标 |
|---|---:|
| 发音提示 P90 | < 3s |
| 语法提示 P90 | < 2.5s |
| 报告生成 | < 10s |

## 9.3 打断策略

| 问题类型 | shouldInterrupt | 展示方式 |
|---|---:|---|
| 轻微发音问题 | false | 浮层提示 |
| 高频发音问题 | false | 对话间隙短提醒 |
| 影响理解 | true | AI 澄清问题 |
| 普通语法错误 | false | 浮层提示 |
| 语义完全不清 | true | AI 简短确认 |
| 系统性问题 | false | 报告总结 |

腾讯智聆采用 WSS：

```text

```

后端生成签名并连接，App 不感知腾讯参数。


`AssessmentConfig`：

```json

  "voiceFormat": 0,
  "evalMode": 3,
```

腾讯智聆关键参数映射：

| 参数 | 推荐值 | 说明 |
|---|---|---|
| `server_engine_type` | `16k_en` | 英语评测 |
| `voice_format` | `0` | PCM |
| `eval_mode` | `3` 或 `1` | 自由说/句子模式，MVP 可按场景选择 |
| `score_coeff` | `3.0-4.0` | 成人严格评分 |
| `sentence_info_enabled` | `1` | 返回中间结果，支撑 UI 更新 |
| `rec_mode` | `0` | 实时识别 |

腾讯返回字段归一化：

| 腾讯字段 | 内部字段 |
|---|---|
| `SuggestedScore` | `overallScore` |

| `PronFluency` | `fluencyScore` |
| `PronCompletion` | `completenessScore` |
| `Words[].Word` | `wordIssues[].word` |
| `Words[].PronAccuracy` | `wordIssues[].score` |
| `Words[].Mbtm/Metm` | `beginMs/endMs` |

# 17. 安全与合规要求

```text
1. App 不保存、不接收腾讯智聆 SecretId/SecretKey、豆包 API Key。
2. 用户音频 URL 必须使用短期签名，默认 24 小时过期。
3. 原始供应商返回 raw_result 存对象存储，只保存引用，避免 DB 膨胀。
4. 用户可删除历史音频、报告、画像。
5. 管理接口必须具备 RBAC 权限和操作审计。
6. 日志中禁止打印 accessToken、refreshToken、供应商密钥、完整音频 base64。
7. WebSocket 建连、结束、供应商调用必须写 requestId/sessionId，方便链路追踪。
```

# 20. 最终接口架构总结

```text
React Native App 只对接后端 REST 与 WebSocket；后端 Realtime Gateway 负责统一编排豆包、腾讯智聆、ASR、TTS、Redis 热记忆与 DB 落库。场景选择、历史、报告、画像走 REST；语音房间走 3 个 WebSocket 通道：主对话通道负责用户与 coach 的正常语音对话，发音测评通道反复返回词级问题、音标和分数，语法纠错通道返回 quickFix、解释和严重程度。前端可以关闭发音或语法提示显示，但后台仍会持续处理并沉淀到报告。首声链路只追求快速可播，发音测评和详细纠错旁路异步返回，最终在报告中汇总沉淀为用户画像和下一轮推荐。
```
